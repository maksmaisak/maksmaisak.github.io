Change: 931747
Date:   2026-01-09 10:32:57
Description:
[UNB-15154]
Making OnInitialize work before BeginPlay even when we spawn actors inside OnInitialize/OnReinitialize of other actors.

There were two cases where OnInitialize can get called after BeginPlay, both because you can't call OnInitialize if you're already inside OnInitialize.
1. When actors are getting spawned on BeginPlay of the world, which is still technically inside the OnInitialize call of the last object that needed to be initialized. That calls NotifyInitializationFinished and inside of that BeginPlay is called on everything, including the actor pool, which spawns a bunch of actors. Inside their OnActorPostInitializeComponents it can't call OnInitialize because we're inside one already so it skips that and goes for BeginPlay, and later, on the next frame it actually does the OnInitialize which triggers an ensure in one of our functional tests.
2. When actors are spawned directly inside OnInitialize/OnReinitialize of an actor: when we reinitialize the worldstate manager we can respawn some actors which should have OnInitialize called but this doesn't happen because we're inside one already.

This change covers both cases. The only caveat is dependencies: if the worldstate manager spawns an actor during OnReinitialize and some component on that (e.g. the worldstate component) has the worldstate manager as an initialization dependency, then the OnInitialize call will still be postponed. This is not ideal but it's still an improvement as it gives guarantees on the OnInitialize<BeginPlay order to other classes outside of this dependency chain.

Vetted by Tim Toxopeus
