Perception

Editor screenshot: select a human AI character, enable "Show > Impact > Show Sight Fields: Of Selected Actor". Show coffin-shaped and cone-shaped sight fields (Focus, Normal, Periphery) with Metro architecture in background.
images/metro_awakening/perception_sight_fields.jpg

Unreal's built-in AIPerception didn't meet our needs for performance or flexibility, so I built a custom perception framework from scratch. The core is sense-agnostic and time-sliced: a central subsystem distributes sensor updates across frames within a configurable millisecond budget, and new sensor types can be added in C++ or Blueprints without modifying the framework.

images/metro_awakening/perception_darkness.mp4

Sight

Each character has multiple sight fields (Focus, Normal, Periphery) with configurable shapes (cones or "coffin" trapezoidal frustums) and priorities. Sight fields detect sight target components on other actors (head, hands, body), and the AI's awareness rate depends on which targets are visible in which fields.

The sight system integrates with a separate darkness system: sight targets report their light level, and enemies cannot detect targets in dark areas unless very close.

Hearing

Noises propagate through the environment using navmesh-based pathfinding with per-material obstacle dampening. A gunshot in a sealed room may not reach enemies two rooms away, but travels far down an open hallway. This makes stealth feel predictable to the player.

Custom Sensors

The framework's extensibility enabled game-specific sensors: a projectile sensor detects bullets flying past a character's head, and dead body detection uses sight targets placed on corpse limbs so bodies sticking out from corners are discovered naturally through the time-sliced sight system.

Perception Debugging

Editor Show Flags visualize sight fields in the viewport. Runtime CVars draw hearing traces and noise propagation paths. Deep Visual Logger integration serializes sight field shapes as custom geometry, enabling full perception replay and debugging after the fact.

Cover System

Editor screenshot: show auto-generated cover points along navmesh edges. Enable cover point visualization, position camera for wide shot showing low cover (one color) and high cover (another) with line-of-fire indicators.
images/metro_awakening/cover_system_overview.jpg

Cover points are automatically generated along navmesh edges when the navmesh is built, requiring no manual placement. Points are classified as low cover (crouch to hide, stand to shoot) or high cover (lean or step out from a wall).

Each point has a body-coverage quality score: an estimate of how much of the character's body is protected. During generation, points below a quality threshold are rejected with retry-based repositioning. The score is also stored on each point and queryable at runtime via EQS, so AI can prefer higher-quality cover when multiple options are available.

Video (10-15s): enable cover point safety visualization and walk the player around a level with enemies. Points near the player and enemies should visibly update (flash/change color) more often than distant ones. Show green (safe) turning red (unsafe) as the player moves to flank.
images/metro_awakening/cover_dynamic_safety.mp4

Dynamic Safety Tracking

Static CoverData (geometry, safe-cone angular ranges) is generated once in the editor. At runtime, CoverTrackers continuously evaluate which points are safe from a given threat. Multiple trackers can run simultaneously on the same cover data: one tracks what's safe for the enemies from the player's last-known position, another tracks what's safe for the player from the enemies (used during Alarmed search to guess where the player might be hiding).

Updates run within a 0.1ms per-frame budget on Quest 2. Each point has a distance-based staleness deadline: nearby points refresh frequently, distant points rarely. A min-heap scheduler pops the stalest points first.

Screenshot of EQS query debug view showing scored cover points (green = high score, red = low). Use the Gameplay Debugger on a human AI running a "find cover" query.

EQS Integration and Level Design Tools

A custom EQS item type for cover points lets designers build cover-picking queries in the visual EQS editor, scoring by safety, body-coverage quality, line-of-fire status, and distance. Standard EQS tests still work transparently on cover items.

Defense Region volumes let level designers control which cover points AI should use in carefully designed encounters. No-Cover volumes suppress generation in specific areas. A cover-aware navigation query filter makes enemies prefer paths through covered areas instead of running across open ground.

Awareness States and HTN

Video (15-20s): enable AI debug info. Record the sequence: Unaware (patrolling), throw object (Investigate), find dead body (Alarmed/Search), spot player (Combat). Also show de-escalation if player hides.
images/metro_awakening/ai_state_machine.mp4

Enemies progress through awareness states driven by perception: Unaware, Investigate, Alarmed, and Combat. Each state drives fundamentally different behavior, from casual patrol to active searching to direct combat. A gunshot triggers investigation toward the noise source. Finding a dead body can jump straight to Alarmed. If the player stays hidden, enemies de-escalate: Combat to Alarmed, or Investigate back to Unaware.

All decision-making within each state is driven by Hierarchical Task Networks (HTN). A master HTN selects the sub-HTN for the current awareness state, and from there the HTN planner builds a full plan covering movement, attacks, communication, and more. HTNs are the real backbone of all enemy AI in the game; the state machine provides the high-level structure for switching between them.

Enemy Behaviors

Video (10-15s): human enemies in cover-based combat. Show popping in/out of cover, shooting, and at least one enemy repositioning when flanked. Multiple humans coordinating.
images/metro_awakening/enemy_human_combat.mp4

Humans

Human enemies use the cover system for cover-based combat: they pick cover using EQS queries, pathfind to it (preferring safe routes), crouch or lean behind obstacles, and pop out to shoot the player. When flanked, they detect the change within seconds and reposition.

An accuracy system controls shooting difficulty: chance-to-hit is driven by time-in-view curves, meaning enemies become more accurate the longer the player stays exposed. Force-hit and force-miss mechanics ensure both fair gameplay and dramatic tension. Each human variant has separately configurable firearm accuracy parameters.

When the target ducks behind cover, humans switch to suppressive fire, shooting at the player's last known position to maintain pressure.

Nosalises

Nosalises are fast mutants that attack in packs. They use the Combat Circle to surround the player: an obstacle-aware spatial assignment system that distributes them into sectors around the target, adapting to walls, corridors, and tight spaces with an optimal-assignment algorithm.

Their attack pipeline follows a telegraph, attack, land sequence with motion warping. Root motion data from attack animations is analyzed to select the correct directional jump, and they dynamically avoid the player's aim direction during approach, switching between walk and gallop gaits.

During investigation, different noise types trigger contextually different reactions: a gunshot causes immediate escalation to combat, while a thrown can prompts cautious investigation.

Video (10-15s): lurkers hiding, performing bounce or drive-by attacks, returning to hiding spots. If possible, kill one to trigger scatter behavior.
images/metro_awakening/enemy_lurker.mp4

Lurkers

Lurkers are agile creatures that hide between attacks, creating tense encounters where the player must constantly watch for movement. They use three attack types: bounce attacks (leaping toward the player from a distance), drive-by attacks (fast slashing runs), and proximity attacks (triggered when the player gets too close to a hiding spot).

Attacks are coordinated in waves with each lurker picking its attack type based on distance, geometry, and position. When an ally is killed, surviving lurkers scatter to new hiding spots. If the player spots a lurker in its hiding spot, it reacts with a startled escape sequence.

Video (10-15s): hunters using hit-and-run tactics. Show hiding behind walls, popping out with blowguns, dodging, pair attacks.
images/metro_awakening/enemy_hunter.mp4

Hunters (Cultists)

Hunters are human cultists that use guerrilla tactics. They find navmesh-edge hiding spots and pop out to fire blowguns before vanishing. Their dodge is purposeful: they dodge closer to their destination and away from the player's aim direction, making them hard to hit while staying goal-oriented.

Hunters coordinate pair attacks, with wall-width-aware exposure checks and ambush spot validation ensuring they only attack from positions where they can actually hit the player. When no good hiding spots remain, they fall back to more aggressive behavior.

Collision Avoidance

Video (8-12s): multiple characters navigating past each other in a corridor or tight space. Enable debug visualization showing candidate avoidance directions and navmesh edge clipping.
images/metro_awakening/rvo_avoidance.mp4

Unreal's built-in RVO (Reciprocal Velocity Obstacles) avoidance assumes characters have directly controllable velocity, which breaks down for root-motion-driven characters where velocity comes from animations. It also has no awareness of navmesh boundaries, causing characters to be steered off walkable surfaces while trying to avoid each other.

I rewrote the engine's UAvoidanceManager to solve both problems:

  • Navmesh-aware candidate directions: the system evaluates evenly-spaced avoidance directions and clips them against navmesh boundaries instead of discarding them, keeping characters on walkable surfaces
  • Animation-velocity awareness: actual root-motion velocity is used for inter-agent calculations instead of desired velocity, so the system accurately predicts where animation-driven characters will be
  • Reciprocal signaling: when one character is stationary, the moving agent takes full avoidance responsibility instead of both agents contributing half (which results in neither avoiding enough)
  • Minimize-penetration fallback: when collision is unavoidable, the system picks the direction that minimizes interpenetration depth

Combat Circle

Top-down or angled screenshot: combat circle debug visualization showing the circle, sectors, obstacles blocking arcs, and agents assigned to sectors. A corridor showing a partial/semicircle is ideal.
images/metro_awakening/combat_circle.jpg

The Combat Circle distributes melee-focused AI (primarily Nosalises) into sectors around the player. A time-sliced obstacle detector incrementally raycasts around the circle, projecting onto navmesh to identify obstacle-free arcs.

An optimal assignment solver maps agents to sectors, minimizing total travel distance while penalizing paths through the circle center (to prevent running through the player) and preferring agents already close. The result: enemies naturally surround the player in open areas and form semicircles near walls, without any manual scripting per encounter.

Damage Response

Video (8-12s): shoot enemies in different body parts. Show directional hit reactions and full stagger when damage threshold is reached.
images/metro_awakening/stagger_hit_reactions.mp4

Two tiers of damage feedback. Hit reactions are additive animations that play on top of current behavior without interrupting it: a flinch, a stumble, minor acknowledgment of being shot. Staggers are full-body animations that interrupt behavior entirely: the character stops what it's doing and plays a stagger.

Incoming damage accumulates with per-bone and per-damage-type multipliers (headshots contribute far more than limb shots). When the stagger threshold is reached, behavior is interrupted. A Hit Reaction Selector data asset maps animations by damage type, hit bone, and direction, letting animators configure all reactions without touching code.

AI Infrastructure

AI Token System

Limits how many enemies can perform a given action simultaneously: at most 2 enemies shoot while 1 lays suppressive fire. Token types include regular fire, suppressive fire, and melee, with configurable cooldowns, lifetimes, and priority-based token stealing so higher-priority agents can take over when needed.

Parameter Templates

A data-driven configuration system with inheritance. A base template (e.g. "Human") defines defaults, and per-character children override only what differs. Properties support Override, Add, and Multiply modes. Changes to a parent propagate instantly to all active children in the world.

AI Groups and Location Reservation

A group coordination framework for encounter management. Includes a location reservation system, integrated with both HTN and EQS, that prevents multiple enemies from rushing to the same cover point or investigate location.

Gameplay Tag Volumes

Level designers place volumes tagged with gameplay tags to control AI behavior spatially: making the player invisible to unaware enemies in specific areas, forbidding certain attack types in tight spaces, or marking valid ambush positions. Stored in an octree for efficient spatial queries, with EQS integration.

Bark System and Voice Lines

A priority-based voice line system with bark groups, cooldowns, and max-simultaneous-barker limits. A voice actor switch Sound Cue node automatically selects the correct voice recording based on the character's template. Bark metadata links to facial animations, automatically playing matching lip sync.

EQS Extensions

A rich library of custom generators (cover points, combat circle sectors, actors-in-range), tests (sight visibility, gameplay tag volumes, collision prediction), and data bindings that allow AI designers to build complex spatial queries entirely in the visual EQS editor.

Sequencer Extensions

Locomotion Tracks let level designers command AI characters to walk along paths using navigation (not canned animation) within Sequencer timelines. Gate Tracks pause sequences until Blueprint-defined conditions are met. A custom sequence player supports dynamic playback conditions and actor participant tracking.

AI Announcements

A decoupled inter-AI messaging system within groups. Enemies share information like last-known enemy locations, loss of visual contact, and calls for help without direct coupling between sender and receiver. Used for squad coordination and contextual barks.

Animation Selection

A root-motion-aware animation selection system. Animators drag-and-drop jump or attack animations into a data asset; at runtime, the system picks whichever animation's root motion best matches the required movement (e.g. closest direction and distance to the target), minimizing the warping needed. Used for Nosalis directional attacks, obstacle jumps via NavLinks, and more.

Actor Pooling

Screenshot: editor showing VGActorPool spawner setup, or a profiler capture demonstrating eliminated spawn hitches, or the Test Automation window with actor pooling tests passing.
images/metro_awakening/actor_pooling.jpg

Characters are pre-spawned during loading and recycled when killed, eliminating multi-millisecond spawn hitches during gameplay, essential for VR's strict frame budget. The system handles complete state reset (perception, avoidance, collision, weapons, animation, navigation) across all enemy archetypes.

Automated functional tests verify that every property is correctly reset on pool return by comparing component state snapshots before and after round-trips. Corpse management with per-platform limits keeps memory under control on Quest 2.

Testing and Optimization

I introduced automated functional testing to the project using Unreal's Test Automation Framework. Tests cover actor pooling, enemy behavior, locomotion track interactions, hit reactions, combat circle assignment, and encounter stress tests that measure per-creature frame-time impact on Quest 2.

Performance optimization work included:

  • Time-slicing across perception, cover updates, combat circle obstacle detection, and hearing events
  • Blueprint-to-C++ ports of hot-path cover and perception code
  • Animation Budget Allocator integration and full fast-path conversion of combat and locomotion Animation Blueprints
  • Cover system safe-cone pre-generation, doubling update throughput
  • 120x path recalculation reduction by fixing stale references that forced unnecessary updates long after paths were no longer needed
